  <action>
Add three new API endpoints to `register_dashboard_routes()` in dashboard.py. Register them BEFORE the /dashboard and /dashboard/{path:path} catch-all routes but AFTER the existing /dashboard/api/* routes, maintaining the route registration order documented in the docstring.

**1. GET /dashboard/api/reviews** — List reviews with optional filtering and sorting.

Query parameters:
- `status` (optional): Filter by review status (pending, claimed, approved, changes_requested, closed)
- `category` (optional): Filter by category (plan_review, code_change, verification, handoff)
- `sort` (optional, default "updated_desc"): Sort order. Values: "updated_desc", "updated_asc", "created_desc", "created_asc", "priority"
- `limit` (optional, default 100): Max results
- `offset` (optional, default 0): Pagination offset

SQL query duplicates the pattern from `get_activity_feed` in tools.py (do NOT import from tools.py — avoid MCP Context dependency). Select from reviews table with subqueries for message_count and last_message_at. Include columns: id, status, intent, agent_type, agent_role, phase, plan, task, priority, project, category, claimed_by, verdict_reason, created_at, updated_at, message_count, last_message_at.

Build WHERE clause dynamically from filter params (same pattern as `_query_review_stats`). Apply ORDER BY based on sort param:
- "updated_desc": `r.updated_at DESC`
- "updated_asc": `r.updated_at ASC`
- "created_desc": `r.created_at DESC`
- "created_asc": `r.created_at ASC`
- "priority": `CASE COALESCE(priority, 'normal') WHEN 'critical' THEN 0 WHEN 'normal' THEN 1 WHEN 'low' THEN 2 END, r.updated_at DESC`

Apply LIMIT and OFFSET. Return JSON: `{"reviews": [...], "total": N}` where total is the unfiltered/filtered count (run a separate COUNT query with same WHERE but no LIMIT/OFFSET).

**2. GET /dashboard/api/reviews/{review_id}** — Full review detail.

Use `{review_id:path}` Starlette route parameter. Query the reviews table for full row (all columns including diff, description, affected_files, counter_patch, counter_patch_affected_files, counter_patch_status). Parse affected_files and counter_patch_affected_files from JSON strings (try/except json.loads like get_proposal in tools.py). Return 404 PlainTextResponse if not found.

Also query audit_events for this review_id, ordered by id ASC. Parse metadata JSON. Return combined JSON:
```json
{
  "review": { ...all review fields... },
  "audit_events": [ { id, event_type, actor, old_status, new_status, metadata, created_at } ]
}
```

**3. GET /dashboard/api/reviews/{review_id}/discussion** — Discussion timeline.

Query messages table for review_id, ordered by rowid ASC. Also query audit_events for verdict_submitted and verdict_comment events for this review_id, ordered by id ASC. Parse metadata JSON on both.

Merge messages and verdict events into a single chronological timeline sorted by created_at. Each item has a `type` field: "message" or "verdict". Message items include: id, sender_role, round, body, metadata, created_at. Verdict items include: id, event_type, actor, old_status, new_status, metadata, created_at.

Return 404 if review not found. Return JSON:
```json
{
  "review_id": "...",
  "timeline": [ { type: "message"|"verdict", ...fields } ],
  "message_count": N,
  "verdict_count": N
}
```

Also add a helper function `_query_reviews_summary(db)` that returns the total count of reviews by status (for the sidebar badge). This will be used by the SSE stream.

Update the `register_dashboard_routes` docstring to include the new route registration order.

**SSE review_update events in dashboard.py:**

Modify the `dashboard_events` SSE endpoint's event_stream generator to also push `review_update` events. Add a review change detection mechanism:

In the SSE loop (alongside the existing log tail and overview_update logic), on each SSE_HEARTBEAT_INTERVAL cycle (reuse the same tick), query `_query_reviews_summary(db)` to get the current review count by status. Compare against a cached previous value. If changed, push a `review_update` SSE event:
```
data: {"type": "review_update", "summary": {"total": N, "by_status": {...}}, "changed_at": "ISO timestamp"}
```

Also push an initial review_update immediately after the initial overview_update on connection (same pattern). This gives the frontend a baseline to detect changes.

The review_update event is lightweight (just counts) — it signals the frontend to re-fetch the full list if the reviews tab is active. This avoids pushing large review payloads over SSE.
